# 아이디어 브레인스토밍 가이드

아이디어를 **적용할지 말지** 고민할 때 아래를 순서 없이 스쳐 지나가며 메모한다. 모두 Yes일 필요는 없다.

## 1. 사용자·가치

- 어떤 화면/운영 시나리오가 좋아지는가?
- “관찰 후보 / 판단 보조” 성격을 해치지 않는가?

## 2. 계약·호환 (additive)

- 기존 응답 필드를 **삭제**해야 하는가? → 삭제가 필요하면 마이그레이션 문서가 있는가?
- 클라이언트·다른 팀이 **옵션 필드**로 소화할 수 있는가?

## 3. 운영·비용

- `web_ops_events` 쓰기·cooldown·budget을 **악화**시키지 않는가?
- read-only API에서 DB write가 필요해지는가? (원칙상 피함)

## 4. 데이터·보안

- `user_key` 스코프 밖 데이터가 노출되지 않는가?
- 로그·qualityMeta에 시크릿·원문 프롬프트가 새지 않는가?

## 5. 검증 방법

- 단위 테스트로 막을 수 있는가?
- 스테이징에서 재현 가능한 시나리오는 무엇인가?

## 6. 롤백

- 기능 플래그·환경변수로 끌 수 있는가?
- 문제 시 사용자에게 보이는 **fallback 문구**는?

## 7. 도메인별 빠른 태그

아이디어 한 줄에 태그를 붙이면 백로그 정렬이 쉽다.

- `#today-brief` `#sector-radar` `#research-center` `#pb` `#ops` `#perf` `#ux`

## 개인화 투자 보조 관점

- 이 아이디어가 사용자의 보유종목·관심종목·투자성향·손실감내를 **더 잘 반영**하는가?
- 사용자에게 **“왜 나에게 이 후보가 중요한지”** 설명할 수 있는가?
- PB / Research Center / Today Brief / Sector Radar 중 **어느 흐름과 연결**되는가?
- **매수 추천처럼 오해**될 문구가 있는가?
- 후보 점수와 리스크 설명이 **분리**되어 있는가?
- 데이터가 부족할 때 **fallback 문구**가 있는가?
