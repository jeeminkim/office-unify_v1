# 목적 · 성공 상태 · 비범위

## 왜 이 문서 세트를 두는가

- 고도화 논의가 **회의록처럼 흩어지지 않게** 하고,
- 아이디어를 **같은 형식**으로 적어 비교·우선순위화하기 위해,
- 구현 전에 **제품 원칙(additive, 관찰 후보, 비자동매매 등)** 과 충돌하는지 빠르게 걸러내기 위해.

## “목적에 가깝다”고 볼 수 있는 상태 (예시)

다음은 팀이 정한 기준에 맞게 조정해도 된다.

- Today Brief / Research Center / PB 후속 흐름이 **문서와 코드 양쪽에서 추적 가능**하다.
- 운영자가 빈 데이터·degraded를 **원인 코드 수준**까지 좁힐 수 있다.
- 새 기능 제안마다 **사용자 가치 / 운영 비용 / 계약 깨짐 위험**이 한 페이지에 정리된다.

## 비범위 (리마인더)

- 자동매매·자동 주문·자동 포트폴리오 변경을 전제로 한 아이디어는 **본 저장소 제품 원칙과 맞지 않으면** 여기서도 “비범위”로 표시하고 종료한다.
- 기존 API/클라이언트 계약을 깨는 변경은 **별도 마이그레이션 계획** 없이 백로그에만 두지 않는다.

개인화 투자 보조 관점에서 아래는 **명시적 비범위**다.

- 자동매매
- 자동 주문
- 사용자 승인 없는 포트폴리오 변경
- 단정적 수익 보장·매수 강요 문구

## 성공 지표 (선택, 정량화 가능하면)

| 지표 | 메모 |
|------|------|
| 예: P95 생성 지연 | 임계값은 인프라와 합의 |
| 예: ops 빈 원인 분류 커버리지 | qualityMeta/ops 코드로 측정 |
| 예: RC follow-up → PB 전환율 | 제품 목표가 있을 때만 |
| Today Brief 후보 설명 가능성 | 사용자가 “왜 이 후보인지” 이해하는 정도(설문·정성 평가 등) |
| RC follow-up → PB 전환율 | follow-up 저장 후 PB 고찰까지 이어진 비율 |
| 미국 신호 empty reason 분류 커버리지 | `primaryReason` 등 진단 코드가 얼마나 안정적으로 채워지는지 |
| PB 고찰 후 사용자 상태 분류율 | 예: 확인함·보류·추가 질문 등(정의는 제품과 합의) |
| 후보 중복 감소율 | 동일 맥락 후보가 덱·목록에서 겹치지 않게 조정 |
| 리스크 경고 확인율 | 상세·경고 노출 후 사용자가 관련 영역을 연 경우 등 |

(숫자는 팀에서 채운다.)
