# Office Unify Project Vision

## 문서 목적
이 문서는 `ai_office`와 `dev_support`를 하나의 통합 서비스로 재구성하는 이유와,
장기적으로 어떤 제품을 만들고자 하는지에 대한 비전을 정리한 문서다.

이 문서는 구현 세부사항보다 상위의 방향성을 다루며,
향후 설계, 구현, 문서화, GPT Builder 확장 시 공통 기준점으로 사용한다.

---

# 1. 프로젝트 배경

현재 두 개의 프로젝트가 별도로 존재한다.

- `ai_office`
- `dev_support`

이 두 프로젝트는 각각 독립적인 목적과 구조를 갖고 발전해왔지만,
실사용 관점에서는 하나의 서비스 안에서 함께 제공될 때 더 큰 가치를 가진다.

기존 구조는 다음과 같은 한계를 가진다.

- 기능이 분산되어 있다.
- 사용자 경험이 분리되어 있다.
- 운영 기준이 일관되지 않다.
- 일부 로직은 특정 채널(예: Discord)에 강하게 결합되어 있다.
- 재사용 가능한 핵심 로직이 제품 단위로 통합되지 못하고 있다.

따라서 이제는 개별 서비스 운영보다,
하나의 통합 웹 앱으로 재구성하는 방향이 더 적절하다.

---

# 2. 프로젝트의 핵심 비전

이 프로젝트의 핵심 비전은 다음과 같다.

**`dev_support`를 최종 제품 본체로 삼고,  
`ai_office`의 핵심 기능과 로직을 점진적으로 흡수하여  
하나의 통합 웹 서비스로 재구성한다.**

이 통합은 단순 병합이 아니다.

목표는 아래와 같다.

- 사용자 입장에서 더 일관된 경험 제공
- 로직 재사용성 향상
- 웹 기반 운영 구조로 전환
- 보안과 서버 경계 강화
- 점진적 확장 가능한 구조 확보
- 향후 AI 도구와의 협업 효율 상승

---

# 3. 최종적으로 만들고 싶은 제품

최종 제품은 단순한 유틸리티 묶음이 아니라,
**하나의 통합 업무/운영/분석 지원 플랫폼**을 지향한다.

예상되는 상위 제품 구조는 아래와 같다.

- 통합 홈
- AI Office
  - 포트폴리오 조회
  - AI 토론
  - 트렌드 분석
  - 운영/상태 확인
- 업무 도우미
  - flow 생성
  - SQL 생성
  - TypeScript/코드 보조
  - 문서/개발 지원 기능

즉, 최종 제품은
“업무 도우미”와 “AI Office”가 따로 존재하는 것이 아니라,
하나의 앱 안에서 역할별 기능으로 연결되는 구조를 지향한다.

---

# 4. 제품 중심축

이 프로젝트의 구조적 중심축은 `dev_support`다.

그 이유는 아래와 같다.

1. 웹 앱 구조로 확장하기에 더 적합하다.
2. 현재 통합의 최종 사용자 경험을 설계하기에 유리하다.
3. UI/레이아웃/정보 구조를 재조직하기 좋다.
4. `ai_office`의 핵심 로직을 서버 계층으로 흡수하기에 적합하다.

따라서 원칙은 다음과 같다.

- 최종 제품 본체는 `dev_support`
- `ai_office`는 기능 원천 및 참조 소스
- 구조의 기준은 `dev_support`
- 기능의 원천 중 일부는 `ai_office`

---

# 5. 통합의 기본 철학

## 5.1 단순 복제가 아니라 재구성
Discord UI를 웹에 그대로 복제하지 않는다.
대신 기능 목적과 사용자 경험을 웹에 맞게 재해석한다.

## 5.2 순수 로직 우선
채널/프레임워크에 묶이지 않은 로직은 최대한 추출해서 재사용한다.

## 5.3 서버 중심 구조
민감정보, 외부 API 호출, 데이터 처리, 판단 로직은 서버 계층으로 이동한다.

## 5.4 기존 기능 보호 우선
새 기능 추가보다 기존 `dev_support` 핵심 기능 안정성이 우선이다.

## 5.5 점진적 통합
모든 것을 한 번에 옮기지 않는다.
작은 단위로 분석, 설계, 구현, 검증을 반복한다.

## 5.6 문서와 검증 포함
구현만으로 완료하지 않는다.
문서, 검증, 리스크 공개까지 포함되어야 한다.

---

# 6. 사용자가 얻는 가치

이 통합이 잘 이루어지면 사용자는 아래 가치를 얻게 된다.

## 6.1 하나의 진입점
여러 프로젝트를 따로 열지 않고, 하나의 앱 안에서 기능을 이용할 수 있다.

## 6.2 일관된 사용자 경험
레이아웃, 네비게이션, 결과 패널, 상태 표시 방식이 통일된다.

## 6.3 더 높은 재사용성
기존 프로젝트의 로직이 더 구조적으로 재사용된다.

## 6.4 더 나은 운영성
Discord에 묶인 운영 흐름 일부를 웹 기반으로 재구성할 수 있다.

## 6.5 향후 확장성
새 기능을 추가할 때 제품 구조가 덜 흔들리고,
AI 협업 도구(Cursor, GPT Builder 등)와의 연결도 쉬워진다.

---

# 7. 왜 하네스 엔지니어링이 필요한가

이 프로젝트는 단순 기능 추가 프로젝트가 아니다.

아래가 동시에 요구된다.

- 기존 두 프로젝트 구조 이해
- 웹 전환 판단
- 서버/클라이언트 경계 관리
- 보안 고려
- 문서화
- 검증
- 점진적 마이그레이션

이런 프로젝트는 프롬프트 하나만 잘 쓴다고 안정적으로 진행되지 않는다.

따라서 이 프로젝트는
**프롬프트 엔지니어링 + 컨텍스트 엔지니어링 + 하네스 엔지니어링**
3층 구조로 운영해야 한다.

### 프롬프트 엔지니어링
현재 턴에서 무엇을 시킬지 명확히 지시한다.

### 컨텍스트 엔지니어링
프로젝트의 배경, 원칙, 구조적 맥락을 항상 유지한다.

### 하네스 엔지니어링
규칙, 계획, 검증, 결과 보고 형식까지 포함한
작업 전체의 레일을 설계한다.

이 프로젝트에서는 특히 하네스 엔지니어링이 중심이다.

---

# 8. 장기 방향성

이 프로젝트는 1차 통합 이후에도 계속 확장될 수 있다.

장기적으로는 아래 방향을 고려할 수 있다.

- 업무 도우미와 AI Office의 더 긴밀한 연결
- 서버 전용 서비스 계층 고도화
- 운영 상태 페이지 확장
- 판단/분석 결과 이력 관리
- 공통 인증/권한 구조
- GPT Builder용 지침/지식 베이스 확장
- 반복 가능한 개발 운영 체계 정착

즉, 이 프로젝트는 단기 통합으로 끝나는 것이 아니라,
**장기적으로 하나의 견고한 제품 체계로 성장하는 기반**을 만드는 작업이다.

---

# 9. 현재 단계에서의 현실적 목표

비전은 크게 가져가되, 현재 단계에서는 현실적인 1차 목표를 유지해야 한다.

현재의 우선 목표는 아래와 같다.

1. 기존 `dev_support` 기능 유지
2. `ai_office`의 웹 전환 가치가 높은 핵심 기능 선별
3. 포트폴리오 조회 / AI 토론 / 트렌드 분석 중심의 1차 통합 설계
4. 서버 계층 흡수 전략 수립
5. 작은 단위의 안전한 구현 준비

### 9.x 위원회 후속작업 운영 보드(Phase 2)

위원회 토론 결과는 사람용 보고서로 끝내지 않고, 운영 가능한 작업 단위로 이어진다.

- 조일현 Markdown(report)과 후속작업(JSON)을 분리
- 저장된 followup만 운영 보드(`/committee-followups`)에서 관리
- 상태 전이는 서버에서 검증(draft→accepted 등)
- 재분석 연결은 payload 준비 단계까지만 제공
- 자동 투자 실행(주문/매매/원장 자동반영)은 금지

### 9.y 위원회 후속작업 재분석 루프(Phase 3)

- 저장된 followup을 기준으로 사용자가 명시적으로 재분석을 실행한다.
- 재분석 결과는 artifact를 append-only로 누적 저장한다.
- completionAssessment(unmet/partial/met)로 완료 기준 충족도를 표현한다.
- 재분석은 작업 판단을 돕는 분석 계층이며, 자동 투자 실행으로 이어지지 않는다.

### 9.z 산업 인포그래픽 생성기(MVP)

- 연구/분석 계열 기능으로 `/infographic`에 분리 배치한다.
- 원문을 LLM이 구조화 JSON으로 정제한 뒤, 고정 템플릿 SVG로 시각 렌더링한다.
- 텍스트/레이블/차트는 코드 렌더링으로 재현성을 확보한다.
- DB 저장은 제외하고 미리보기/PNG 저장까지 제공한다.
- 숫자 추정 생성은 금지하고, 부족 데이터는 `unknown/null/empty`로 처리한다.
- 입력 소스는 text/url/pdf로 확장하되, 모바일 읽기 뷰와 export 저장 뷰를 분리해 실사용 가독성을 우선한다.
- URL/PDF 입력은 추출 텍스트 preview/edit 단계를 거쳐 품질을 사용자가 통제할 수 있게 한다.
- 추출 텍스트는 cleanup 단계를 거쳐 신뢰 가능한 초안을 제공하고, 모바일에서는 읽기/저장 미리보기 UX를 분리한다.
- 품질 메타(debug)와 사용자용 품질 요약(strip)을 병행해 결과의 설명 가능성과 운영 피로도 감소를 함께 달성한다.

즉, 지금 단계는 “완성형 제품”이 아니라,
**올바른 구조로 가기 위한 첫 번째 안정화 단계**다.

# 10. 이 문서의 역할

이 문서는 아래 상황에서 기준점이 된다.

- 왜 통합하는지 설명할 때
- 설계 방향이 흔들릴 때
- 범위가 과도하게 커질 때
- GPT Builder용 상위 지침을 만들 때
- Cursor 작업의 장기 방향을 점검할 때

즉, 이 문서는 기술 문서라기보다
프로젝트의 방향과 존재 이유를 고정하는 비전 문서다.

---

# 11. 결론

이 프로젝트의 목표는 두 프로젝트를 억지로 합치는 것이 아니다.

목표는 다음과 같다.

**`dev_support`를 중심으로 `ai_office`의 핵심 가치를 흡수하여,  
하나의 웹 기반 통합 서비스로 재구성하는 것.**

그리고 그 과정은
즉흥적인 프롬프트가 아니라,
**규칙·계획·검증·문서화를 포함한 하네스 엔지니어링 방식**으로 진행한다.

이 문서는 그 방향의 최상위 비전 기준점이다.
